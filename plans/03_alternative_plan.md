# План решения задачи: Предсказание статических BEV-карт по мультикамерным изображениям

Привет! Задача предсказания статических препятствий (BEV Occupancy / Semantic Segmentation) с нескольких камер — это классическая и очень популярная задача в автономном вождении. Топ-1 скор 0.56 вполне реально побить, если исправить фундаментальные ошибки бейзлайна и применить современные подходы.

Ниже представлен подробный ресерч, разбор ошибок бейзлайна, SOTA-решения, а также фишки и хаки, которые помогут выиграть хакатон с учетом твоих ограничений по железу.

---

## 1. Разбор текущего бейзлайна (Почему метрика всего 0.4?)

В `baseline_v4.ipynb` есть несколько критических проблем, исправление которых сразу даст огромный буст:

1. **Используется только 1 камера из 4-х**: В цикле обучения берется только `images[0]` (фронтальная камера), а остальные 3 камеры просто игнорируются. Модель физически не видит, что происходит по бокам, но пытается предсказать карту вокруг всей машины.
2. **Игнорирование геометрии (матриц)**: Матрицы `intrinsics` (внутренние параметры) и `car_to_cam` (внешние параметры) загружаются в датасете, но никак не используются в `BaseModel`.
3. **Неэффективная архитектура**: Модель использует `view(batch_size, -1)` и `Linear` слой для перевода фичей из 2D в BEV. Это полностью уничтожает пространственную информацию (spatial priors) и требует огромного числа параметров.
4. **Неправильный Loss**: Используется `MSELoss`. Но у нас задача бинарной сегментации (0 - свободно, 1 - занято, 255 - неизвестно). `MSELoss` плохо штрафует за ошибки классификации.
5. **Фатальная ошибка с Intrinsics при Resize**: В датасете картинки ресайзятся с оригинального размера (1024x542) до (512x256). **При изменении размера картинки необходимо пропорционально изменять матрицу intrinsics!** Бейзлайн этого не делает. Если ты начнешь использовать матрицы с текущим ресайзом, проекция будет кривой.

## 2. SOTA решения для BEV (State-of-the-Art)

Для перевода изображений с нескольких камер в единый вид сверху (BEV) сейчас доминируют три подхода. Все они решают проблему "View Transformation" (трансформация вида).

### A. Lift, Splat, Shoot (LSS)
**Как работает:** Для каждого пикселя картинки модель предсказывает распределение его глубины (Lift). Зная глубину и луч (из intrinsics), пиксель превращается в 3D-точку. Затем все точки со всех 4-х камер переводятся в систему координат машины (через `car_to_cam`) и "сплющиваются" на 2D BEV-сетку (Splat).
**Плюсы:** Очень быстрый, хорошо работает для статики.
**Где взять:** [Оригинальный репозиторий LSS](https://github.com/nv-tlabs/lift-splat-shoot)

### B. Cross-View Transformers (CVT)
**Как работает:** Создается пустая BEV-сетка (query). С помощью механизма Cross-Attention и матриц камер, каждая ячейка BEV-сетки "смотрит" на нужные пиксели с 4-х камер и забирает оттуда признаки.
**Плюсы:** Идеально для статических препятствий, не требует явного предсказания глубины, отлично работает с неточными калибровками.
**Где взять:** [bradyz/cross_view_transformers](https://github.com/bradyz/cross_view_transformers) — **Настоятельно рекомендую этот репозиторий для хакатона.**

### C. Simple-BEV
**Как работает:** Создается 3D воксельная сетка вокруг машины. Каждый воксель жестко проецируется на 4 камеры (используя матрицы). Фичи с камер билинейно интерполируются в эти воксели. Затем ось Z (высота) сплющивается свертками.
**Плюсы:** Невероятно прост в реализации (можно написать на чистом PyTorch за 100 строк), работает лучше LSS.
**Где взять:** [aharley/simple_bev](https://github.com/aharley/simple_bev)

## 3. План действий и архитектура (под M1 Pro / T4 / V100)

Учитывая ограничение по вычислительным ресурсам (59ч T4 или 18ч A100), обучать тяжелые трансформеры типа BEVFormer с нуля не выйдет. Нужно собирать эффективную модель.

### Рекомендуемая архитектура (Simple-BEV или CVT-light):
1. **Backbone (Извлечение фичей 2D)**:
   - **Не пиши свертки с нуля!** Используй предобученные модели из `torchvision` или `timm`.
   - Идеальный выбор для T4/M1: `resnet34`, `resnet50` или `efficientnet_b0`. Они быстрые и дают отличные фичи.
   - Прогоняешь 4 картинки через Backbone $\rightarrow$ получаешь 4 карты признаков (feature maps).
2. **View Transformer (Перевод в BEV)**:
   - Реализуй подход **IPM (Inverse Perspective Mapping)** или **Simple-BEV**.
   - Для IPM: так как дорога плоская, можно вычислить матрицу гомографии (Homography) для каждой камеры и с помощью `torch.nn.functional.grid_sample` "натянуть" 2D фичи на BEV-плоскость.
   - Складываешь фичи с 4-х камер в один BEV-тензор.
3. **BEV Decoder (Сегментация)**:
   - Обычный UNet-декодер или FPN (Feature Pyramid Network), состоящий из 2D сверток (`Conv2d`, `BatchNorm`, `ReLU`), который принимает BEV-тензор и выдает маску размером `(1, 188, 126)`.

## 4. Фишки и хаки для победы (Как пробить 0.56+)

Эти трюки часто решают исход соревнований по BEV-сегментации:

### 1. Правильный Loss (Критично!)
Замени `MSELoss` на комбинацию `BCEWithLogitsLoss` + `DiceLoss` или `FocalLoss`.
В датасете сильный дисбаланс классов (нулей намного больше, чем единиц).
```python
# Пример расчета Loss
loss_mask = (gt != 255) # Игнорируем неразмеченные зоны
preds_masked = preds[loss_mask]
gt_masked = gt[loss_mask].float()

# Focal Loss или BCE с весами для класса '1' сильно поднимет IoU!
bce_loss = F.binary_cross_entropy_with_logits(preds_masked, gt_masked, pos_weight=torch.tensor([3.0]))
```

### 2. Корректировка Intrinsics при Resize (Критично!)
Если ты ресайзишь картинку с `(1024, 542)` до `(512, 256)`, ты обязан изменить матрицу `intrinsics`:
```python
scale_x = 512 / 1024
scale_y = 256 / 542
intrinsics[0, 0] *= scale_x # f_x
intrinsics[0, 2] *= scale_x # c_x
intrinsics[1, 1] *= scale_y # f_y
intrinsics[1, 2] *= scale_y # c_y
```
Без этого геометрия сломается, и проекция с камер будет указывать в неверные пиксели BEV-карты.

### 3. Аугментации (Data Augmentation)
- **Image-level**: `ColorJitter`, `RandomBrightnessContrast`. Модели часто переобучаются на освещение.
- **Camera Dropout**: Случайно зануляй фичи с 1 из 4 камер во время обучения. Это заставит модель лучше понимать контекст и не полагаться только на фронтальную камеру.
- **BEV-level Augmentation**: Случайные повороты (Random Rotate) и сдвиги (Random Translate) итогового GT BEV. **Важно:** при этом нужно умножить матрицы `car_to_cam` на обратную матрицу этого поворота/сдвига, чтобы камеры "повернулись" вместе с GT.

### 4. Использование Temporal (Исторических) кадров
В датасете есть `timestamp` и `id автомобиля`. Скорее всего, кадры идут последовательно.
Если добавить в модель фичи с предыдущего кадра (t-1), выровняв их по эго-движению машины, метрика взлетит в космос. (Это продвинутый уровень, делай, если останется время).

### 5. Ансамблирование и TTA (Test Time Augmentation)
- **TTA**: На этапе инференса прогоняй картинки как есть, а затем отзеркаленные по горизонтали (Flip). Усредняй логиты.
- **Ансамбль**: Обучи одну модель с `ResNet-34`, а вторую с `EfficientNet-b0`. Усреднение их предсказаний `(pred1 + pred2) / 2` обычно дает +0.02 к IoU бесплатно.

## 5. Как организовать процесс обучения
1. **Локально на M1 Pro (MPS)**: Напиши код, убедись, что модель прогоняет `forward` и `backward` без ошибок. Обучи на 100 семплах (overfit on batch), чтобы проверить, что Loss падает.
2. **Удаленно на T4/V100/A100**: Запускай полноценное обучение. Сохраняй чекпоинты каждую эпоху и валидируй по метрике **IoU** (а не по Loss).
3. **Выбор GPU**: 18 часов на A100 — это очень много для такой задачи. A100 обучит ResNet-34 + SimpleBEV за пару часов до сходимости. Лучше использовать T4 для экспериментов, а A100 оставить на финальные прогоны тяжелых моделей или ансамблей.

Удачи на хакатоне! Если нужна будет помощь с написанием конкретного куска кода (например, View Transformation или Focal Loss) — обращайся!
